# 需求：根治登录闭环产生的 `xhs_account_<时间戳>` 占位废账号

- 日期：2026-07-22
- 提出方：NBDpsy 管理后台侧（运营反馈 + 生产取证）
- 状态：待设计/实施（本文档只定义问题与验收，设计决策留给实施者）

## 1. 现象

运营通过 chrome 插件扫码登录小红书账号时，`xhs_accounts` 表会偶发多出一条名为
`xhs_account_<毫秒时间戳>` 的占位账号行：无 nickname、无 user_id、无 red_id，
cookie 往往随后被判 invalid。它会出现在管理后台的可授权账号列表里（乱码观感），
且此前无法删除（管理后台删除入口 2026-07-22 已补，见 §4）。

## 2. 生产取证（2026-07-22 从 data/nbdpsy.db 取证）

两次真实案例，模式完全一致——**占位行创建后 40~90 秒，同一操作者的真账号登录成功**：

| 占位行 | 创建时间 | 紧随其后的真账号 |
|---|---|---|
| id=3 `xhs_account_1784606714415`（cookie 3492B，user_id NULL） | 07-21 04:05:16 | id=2 NBDpsy聊心理 last_login_at 04:05:57（重登成功） |
| id=4 `xhs_account_1784687069847`（cookie 1764B，后验 invalid） | 07-22 02:24:30 | id=5 生意经 02:25:51 创建成功 |

两条占位行均无任何 publish_jobs / note_metrics 关联，各挂 1 条 operator_account_access
授权行。两行已于 2026-07-22 经管理后台删除（DELETE /api/accounts/{id} 级联清授权）。

## 3. 根因链（代码已定位）

1. **插件侧** `chrome-extension/background/service-worker.js`：
   - 登录检测成功后进个人主页采 userInfo，最多 60 次 × 2s（约 2 分钟）；遇验证码
     /未跳到 profile 页/content script 未就绪则拿不到 nickname，超时后仅 warn 放弃
     （约 345 行）。
   - 推送时 accountName 兜底链（约 420 行）：
     `nickname → xhs_${user_id} → xhs_account_${Date.now()}`，且 user_info 传 null。
2. **服务端** `app/services/cookie_service.py::import_cookies`：
   - upsert 匹配顺序 = user_id → account_name。占位推送两个键都是空/全新时间戳名
     → **必然新建一行**，并给导入者建授权。
3. 操作者随后重试登录成功 → 真账号行正常创建/更新 → 占位行成为永久孤儿。

即：占位行 = 「userInfo 采集失败但 cookie 已采到」这次半成功推送的落库产物，
每一个占位行都对应一次登录闭环里的 userInfo 采集失败。

## 4. 已有的兜底（不属于本需求，仅背景）

NBDpsy 管理后台 2026-07-22（commit 535068c8）已上线账号删除：apikeys 页账号 chip
带 × → `DELETE /api/admin/blog/toolkit/xhs-mcp/accounts/:id` → 本服务
`DELETE /api/accounts/{id}`。运营可以手动清，但属于事后人工兜底，不解决产生机制。

## 5. 需求目标

正常的「失败后重试成功」登录闭环结束后，**系统内不应留下占位废行**；同时**不得
丢失可能有效的 cookie**（占位推送时 cookie 是当次采集的唯一副本，若操作者不再重试，
一刀切拒收会把可能有效的登录成果扔掉——注意 §2 中 id=3 的 cookie 有 3492B，未必无效）。

## 6. 方案方向（建议，不锁死；实施者可另行设计后与用户确认）

按优先级建议组合：

- **方向 A（推荐，服务端自愈）**：import_cookies 走「命中既有号更新」或「新建带
  user_id 的号」成功时，顺带清理**同一 operator** 近窗（如 30 分钟）内创建的、
  `user_id IS NULL AND name LIKE 'xhs_account_%'` 的占位行及其授权行。理由：真登录
  成功即宣告此前的半成功推送作废，占位 cookie 已无保留价值。
- **方向 B（TTL 兜底）**：占位行（同上判据）超过如 24h 仍未回填 user_id 的，由后台
  任务清理。覆盖「操作者失败后当天没有重试」的残留。
- **方向 C（插件侧降噪，可选）**：userInfo 全空时插件在 popup 明示「本次未识别账号
  身份，已暂存 cookie，请重试登录以完成识别」，让操作者知道该重试，而不是静默。
- 不建议：完全禁止占位推送（丢 cookie，违反 §5）；仅靠 cookie_check invalid 清理
  （id=3 的 cookie 未必判 invalid，覆盖不全）。

## 7. 验收标准

1. 复现 §2 场景（首次采集 userInfo 失败 → 占位推送 → 60~90 秒后重试成功）：闭环结束后
   `xhs_accounts` 无 `xhs_account_%` 且 user_id IS NULL 的行残留，其授权行同步清除。
2. 占位行清理**只限定**同 operator + 判据 `user_id IS NULL AND name LIKE 'xhs_account_%'`，
   绝不触碰任何带 user_id 或人工改过名的账号行（PATCH 改名后的行 name 不再匹配前缀，
   天然豁免，需测试覆盖）。
3. 操作者只失败不重试的场景：占位行在 TTL 内仍可见可用（cookie 可下发注入），TTL 过后
   自动清理（若采纳方向 B）。
4. 并发安全：两个 operator 同时登录各自账号，互不误删对方的占位行。
5. 现有测试全绿；新增用例覆盖上述 1/2/4。
